# 概述

UML是一种语言，它遵循特定的规则，允许创建各种模型而并不告诉设计者需要创建哪些模型；而且不提供开发过程。是可视化、图形化语言，用于构造系统或理解系统的语言

## 组成 

```
UML的组成：事物、图和关系

UML的事物：构件事物(类、接口、协作、用例、构件、节点等)，行为事物（交互、状态机），分组事物，注释事物

UML的关系：依赖、关联、泛化、实现

UML的视图：用例视图、逻辑视图、并发视图、组件视图、配置视图

UML的图：用例图、类图、对象图、状态机图、活动图、顺序图、通信图、构件图、部署图、包图、组合结构图、交互概览图、时间图

```

##系统开发

```
需求分析:了解客户的需求

系统分析:分析系统的可行性，分析需求的一致性及正确性等

系统设计:架构设计，模块间的接口设计，数据库设计，算法设计，数据结构设计

程序实现：通过编程语言，将说设计的内容转化为可执行的软件系统

测试阶段：单元测试，集成测试，系统测试，验收测试

```

# 面向对象

## 对象

```
对象是面向对象的基本构造单元，是系统中用来描述客观事物的一个实体，一个对象由一组属性和对属性的进行操作的一组方法组成

对象不仅能表示具体的实体，也能表示抽象的规则、计划或事件：
1.有形的实体：指一切看得见，摸得着的实物
2.作用：指人或组织等所起的作用
3.事件：在特定时间所发生的的事
4.性能说明：制造厂或企业，需对产品的性能进行全面的说明

对象的特征：
模块性：对象是一个独立存在的实体
继承：利用已有的定义作为基础来创建新的定义，而不是重复定义他们
动态链接性：各个独享之间是通过传递消息来建立起连接
```

## 特性

```
封装(Encapsulation)
封装就是把一个对象的方法和属性组成一个独立的单位，并尽可能隐蔽对象的属性、方法和实现细节的过程，仅仅将接口进行对外公开

继承(Inheritance)
继承事一种一般类和特殊类的层次模型。继承性指特殊类的对象具有其一般类的属性和方法，在其上又增加了自己的特殊属性和方法。

多态(Polymorphism)
多态性指类中通一函数名对应多个功能相似的不同函数，可以使用相同的调用方式来调用这些具有不同功能的同名函数，这些同名的函数可以是参数的个数或类型不同，但是函数名相同，当进行调用的时候，根据所传的数据选定相应的函数，从而去执行不同的功能。

消息
消息是实现对象之间进行通信的一种机制，对于一个对象可以接收不同形式的多个消息，并产生不同的结果，相同形式的消息可以发送给不同的队形，并产生不同的结果；在发送消息的时候可以不考虑具体的接收者，对象可以对详细做出相应，也可以拒绝消息。
```

## 分析方法

```
1.识别并筛选对象
按照对象的定义，对象应是实际问题域中有意义的个体或概念实体。对象具有目标软件系统所关心的属性。并且对象应以某种方式与系统发生关联，即对象必须与系统中其他有意义的对象进行消息传递，并提供外部服务
通过对用户需求文档的分析可以找出所有的名词或名词短语，合并同义词，这些事极有可能成为对象的。出去具有动作含义的名词，这些动词将被描述为对象的操作而不是对象本身

2.标识对象的属性
属性是对问题域中对象性质的一个描述，对象在系统中所有可能的状态就是属性的取值。对象一般具有多属性，但在分析阶段就哟啊分析出对象的哪些属性时和系统紧密相关的。
在识别属性过程中，对于某个实体，不仅要求其取值有意义，而且其本省在系统中必须是独立存在，就应该将该实体作为一个对象，而不能作为另一对象的属性。此外，为保持需求模型的简洁性，一般将省略对象的一些导出属性。

3.识别对象的行为
对象的行为可以简单理解为对象对外所提供的所有功能。
一般将对象的行为分为三类：
1）对象声明周期中的创建、维护、删除行为
2）计算型行为
3）监视型或称响应行为
```

# 用例图

## 概述

```
用例模型的基本组成部分有用例、角色(参与者)和系统。用例用于描述系统的功能，也就是从用户的角度来说，系统具有应包含的功能，帮助分析人员理解系统的行为，它是对系统功能的宏观的、整体的描述，一个完整的系统包含许多用例，每个用例具体说明应完成的功能；参与者是指那些与系统进行交互的外部实体，通常它是系统的一个用户，但它也可是其他系统或硬件设备；用例常常必须向参与者传递一些数值，这些数值是参与者在系统中获得的信息

用例图的主要作用：
1）用来描述将要开发系统的功能需求和系统的使用场景
2）作为设计和开发过程的基础，促进各阶段开发工作的进展
3）用于验证与确认系统需求

用例图组成元素：
1）参与者(Actor):系统的用户
2）系统边界(Systerm Scope)：确定系统的范围
3）用例(Use Case)：系统提供的服务
4）关联(Association)：参与者与用例间的关系
```

## 参与者

```
参与者是系统外部的一个人或者事物，它以某种方式参与了系统的执行过程。参与者不是特指人，是指系统以外的，在使用系统或与系统给交互中所扮演的角色。

参与者在UML中通常以一个直立人的图形符号来表示

参与者实际上就是类，因此参与者之间也具有继承关系（但在分析设计阶段，一般用泛化关系表示继承关系）。参与者之间的泛化关系表示一个一般性的参与者(父参与者)与另一个较为特殊的参与者(子参与者)之间的联系。

参与者之间的泛化关系用带空心三角形箭头的实线表示，箭头指向超类（父类）
```

## 用例

```
用例是代表系统中各个项目相关人员之间根据系统的行为所达成的契约。用例描述了在不同条件下，针对某一项目相关人员的请求，系统对其作出的响应。也就是说用例值得是对一组动作的描述，系统通过执行这些动作将对用例的参与者产生可以看到的结果，用来描述参与者可以感受都的系统服务或功能

用例在UML中通常用一个椭圆图形符号来表示

用例在系统的需求分析时具有的特点：
1）用例是从系统的使用角度描述系统中的信息，即在系统的外部所能看到的系统的功能，而不是考虑系统内部对该功能的具体实现方式
2）用例描述了用户提出的一些可见需求，对应一个具体的用户目标。
3）用例通常由摸个参与者来执行
4）用例把执行的结果反馈给参与者
5）用例咋功能上具有完整性，即它从参与者接输入，产生的结果最终再输出给参与者
```

### 用例之间关系

- 包含关系

```1）包含关系
指的是两个用例之间的关系，其中一个用例的行为包含另一个用例的行为。也就是说基本用例会用到包含用例，表示基本用例中重用包含用例中的步骤

在UML中，使用带虚线箭头表示，并在线上标有<<nclude>>，箭头的方向是从基本用例到包含用例
```

- 扩展关系

```
扩展关系是对基本用例的扩展，基本用例是一个完整用例，即使没有字用例的参与，也可以完成一个完整的功能。扩展的基本用例中将存在一个扩展点，只有当扩展点被激活时，子用例才会被执行。在扩展关系中，基本用例必须声明若干“扩展点”,而扩展用例只能在扩展点上增加新的行为和含义。扩展关系是从扩展用例到基本用例的关系，它说明扩展用例定义的行为如何插入到基本用例定义的行为中。扩展用例并不在基本用例中显示

扩展用例使用情况
1）表明用例的某一部分是可选的系统行为
2）表名只在特定条件下才执行的分支
3）表名可能有一组行为，其中的一个或多个可以在基本用例中的扩展点插入，所插入的行为和插入的顺序取决于在执行基本用例时与主角进行的交互

在UML中，使用带虚线箭头表示，并在线上表有<<extend>>,箭头方向是从扩展用例指向基本用例
```
